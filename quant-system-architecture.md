## 整体架构：1+1 模式

微服务维护成本高、问题定位难，不适合小团队低频系统。
采用**2个进程**的精简架构：数据服务 + 主服务，通过数据库解耦，无需服务间直接通信。

> **行业验证**：调研 Freqtrade、Hummingbot、Jesse、BBGO、GoCryptoTrader 等主流量化框架，
> 无一例外采用单体或模块化单体架构，没有使用微服务的先例。

```
┌─────────────────────────────────────────────────────┐
│              Vue 3 前端（监控看板）                   │
│        持仓概览 / 信号日志 / 盈亏曲线                 │
└──────────────────────┬──────────────────────────────┘
                       │ REST + WebSocket
┌──────────────────────▼──────────────────────────────┐
│                  主服务（Go）                         │
│                                                     │
│  定时调度 → 指标计算 → R1分析 → 风控校验 → 下单执行   │
│                                                     │
│           Binance Futures API（下单/查仓）            │
└──────────────────────┬──────────────────────────────┘
                       │ 只读
                  MySQL + Redis
                       │ 只写
┌──────────────────────▼──────────────────────────────┐
│                  数据服务（Go）                       │
│                                                     │
│  Binance WebSocket → Top20 实时行情 → Redis          │
│  Binance REST      → 历史K线        → MySQL          │
│  CryptoPanic API   → 新闻抓取       → MySQL          │
│  Alternative.me    → 恐贪指数       → Redis          │
│  Binance REST      → 资金费率       → MySQL          │
└─────────────────────────────────────────────────────┘
```

### 架构参考：BBGO 双流模式

BBGO（Go 量化框架，GitHub 1.6k stars）采用的双流架构与我们天然对应：

```
BBGO 双流                        我们的 1+1
─────────────────                ─────────────────
MarketDataStream（只读）    ←→   数据服务（只写库）
  ├─ K线                           ├─ K线 → MySQL
  ├─ 公开成交                       ├─ 实时行情 → Redis
  └─ 行情数据                       └─ 新闻/恐贪 → MySQL

UserDataStream（账户）      ←→   主服务（读库+决策+执行）
  ├─ 订单更新                       ├─ 指标计算
  ├─ 成交回报                       ├─ R1 分析
  └─ 余额变动                       └─ 风控 + 下单
```

## 服务边界与解耦

### 逻辑边界

**数据服务**：只负责收集和存储，不关心数据如何被使用
- 维护 Binance WebSocket 长连接（20+ 标的）
- 拉取历史 K线 → MySQL
- 轮询新闻/恐贪指数 → MySQL / Redis
- 采集资金费率 → MySQL
- 不与主服务直接通信

**主服务**：只负责读取和决策，不关心数据从哪来
- 从 MySQL 读 K线 → 计算技术指标
- 从 Redis 读实时价格
- 从 MySQL 读新闻 → 构建 Prompt → 调用 R1
- 从 MySQL 读资金费率 → 纳入 R1 分析和风控
- 风控校验 → 调用 Binance API 下单

### 解耦方式

两个服务通过 MySQL + Redis 共享数据，**无需 HTTP/RPC/消息队列**直接通信。

```
数据服务                         主服务
   │── 写 MySQL（K线/新闻）  ──→ │── 读 MySQL
   │── 写 Redis（实时价格）  ──→ │── 读 Redis
   └── 互不感知，独立运行         ┘
```

触发机制：数据服务持续写入，主服务每日定时启动读库，数据必然已就绪。
日线低频场景下，定时触发足够可靠，无需事件通知机制。

## 每日自动执行链路

```
每日定时触发（如收盘后）
    ↓
数据服务已持续维护 Top20 实时数据
    ↓
主服务：拉历史K线 → 计算技术指标（RSI/MACD/BB/EMA）
    ↓
主服务：读新闻摘要 + 恐贪指数 + 资金费率
    ↓
主服务：构建 Prompt → 调用 DeepSeek R1
    ↓
R1 输出 JSON：{标的, 方向, 杠杆建议, 仓位比例, 止损止盈}
    ↓
风控硬校验：杠杆 ≤ 10x / 单仓 ≤ 20% / 总敞口 ≤ 50% / 距强平 ≥ 15%
    ↓
调用 Binance Futures API 下单
    ↓
结果写 MySQL → Redis 推送 → 前端展示
```

## 行业调研：主流框架对比

### 框架概览

| | Freqtrade | Hummingbot | Jesse | BBGO | GoCryptoTrader |
|--|-----------|------------|-------|------|----------------|
| 语言 | Python | Python | Python | **Go** | **Go** |
| 架构 | 模块化单体 | 多组件 | 分层 | 流式双通道 | 引擎+调度器 |
| 策略定义 | IStrategy 类 | Script/Controller | IStrategy 类 | 组件注入 | 插件式 |
| 执行模型 | 5s 轮询 | 1s 时钟驱动 | K线驱动 | 事件流 | 事件驱动 |
| 交易所对接 | ccxt | 自研连接器 | 自研 | 自研+WS | 自研 |
| 回测 | 有 | 有 | 有 | 有 | 有 |
| Stars | 极高 | 高 | 中 | ~1.6k | ~3.4k |

### 各框架核心设计

**Freqtrade** — 模块化单体标杆
- IStrategy 抽象类：populate_indicators() → populate_entry_trend() → populate_exit_trend()
- SQLAlchemy 持久化 Trade/Order 对象
- FreqAI 模块异步运行 ML 预测，不阻塞交易主流程
- 默认 5s 轮询循环，回测时按 K线执行

**Hummingbot** — 时钟驱动多组件
- 独特的 Clock-Based 事件循环（默认 1s tick）
- V2 框架：Script → Controller → Executor 三层
- 连接器标准化接口（REST + WebSocket 双通道，自动降级）
- Dashboard + API 支持多机器人编排

**Jesse** — 统一回测/实盘代码
- 一套策略代码同时运行在回测、模拟、实盘三种模式
- 300+ 内置指标，NumPy 向量化计算
- 分钟级回测引擎，多时间框架同步

**BBGO** — Go 双流架构
- MarketDataStream + UserDataStream 分离
- 组件注入模式，策略接收注入的服务依赖
- Order Executor API 隔离策略间的订单管理

**GoCryptoTrader** — 引擎+调度器
- 中央 Engine 协调所有子系统
- Dispatcher Pub/Sub 模式处理子系统间通信
- 独立的技术分析库 gct-ta

### 共同设计模式

所有框架不论语言，核心流程一致：

```
加载行情 → 计算指标 → 生成信号 → 风控校验 → 执行下单 → 记录结果
```

**订单状态机**（所有框架标配）：

```
Created → Submitted → Accepted → Working
                                    ↓
                          Partially Filled → Cancelled
                          Fully Filled（终态）
```

**WebSocket 重连策略**（通用模式）：
- 心跳保活（25~30s ping/pong）
- 指数退避重连（1s → 2s → 4s → 8s → 上限）
- 重连后恢复全部订阅
- REST 回填重连期间遗漏数据

## 交易策略调研

### 适合本系统的策略组合

| 策略 | 优先级 | 说明 |
|------|--------|------|
| **趋势跟踪** | ⭐⭐⭐ 主策略 | MA 交叉/突破，日线/周线天然适合趋势跟踪 |
| **资金费率监控** | ⭐⭐⭐ 辅助信号 | 永续合约特有；持续正费率可能是拥挤预警而非趋势确认 |
| **多时间框架分析** | ⭐⭐ 增强 | R1 可同时分析日线趋势 + 周线格局 |
| 均值回归 | ⭐ 补充 | 震荡市中与趋势互补，R1 可根据市场状态切换 |
| 网格交易 | 不采用 | 需高频操作，不适合日线级别 |
| 资金费率套利 | 待评估 | 需同时持有现货+合约，架构复杂度高 |

### 资金费率的重要性

永续合约独有机制，每 8 小时结算一次：
- **正费率**：多头付空头（做多有成本）
- **负费率**：空头付多头（做空有成本）
- 2025 基准：平均 0.015%/8h，年化约 19.26%

**对系统的影响**：
- 长期持仓时资金费率会侵蚀利润
- 极端费率是市场拥挤的预警信号
- R1 分析需要将资金费率纳入决策因素

### 2025 年行业教训

| 教训 | 对本系统的启示 |
|------|--------------|
| 过度杠杆是头号杀手 | 硬编码杠杆上限不可商量 |
| 资金费率被误读 | 持续正费率 ≠ 趋势强，可能是拥挤预警；R1 需理解此信号 |
| 回测 ≠ 实盘 | 回测必须建模滑点（0.1~0.3%）和手续费（0.02~0.04%） |
| 闪崩无预警 | 必须设日亏损熔断机制 |
| 加密资产高度相关 | BTC 跌全部跌，多标的 ≠ 分散风险；需监控相关性 |
| 回测数据存在幸存者偏差 | 历史数据需交叉验证，退市标的不可忽略 |

## 语言选型结论

### 团队技术背景
Go、PHP、Java、Python、Node.js、MySQL、Redis、Docker

### 最终决策：数据服务 + 主服务 均用 Go ✅

| 决策点 | 结论 |
|--------|------|
| 数据服务语言 | Go |
| 主服务语言 | Go |
| Binance 签名封装 | 手写 HMAC-SHA256 封装（团队接受），可参考 adshao/go-binance 实现 |
| 指标计算库 | cinar/indicator |
| 统一技术栈 | 是，只维护一套语言 |

### 选型理由

- 团队熟悉 Go，无额外学习成本
- goroutine 天然适合 20+ WebSocket 长连接并发维护
- cinar/indicator 覆盖全部所需指标（RSI/MACD/BB/EMA）
- 一套技术栈：统一日志、统一部署、统一问题排查
- 编译型语言，线上问题更容易复现和定位
- 手写 Binance HMAC 签名封装约 1~2 天工作量，之后作为内部稳定库复用

### 行业验证

BBGO（~1.6k stars）和 GoCryptoTrader（~3.4k stars）两个生产级 Go 量化框架证明 Go 技术栈完全可行。

### Go 依赖库清单

| 用途 | 库 | 说明 |
|------|----|------|
| Binance API | adshao/go-binance/v2 | Futures 完整支持，内置 HMAC 签名 |
| 技术指标 | cinar/indicator/v2 | 零依赖，流式处理，支持泛型，自带回测 |
| 定时任务 | robfig/cron/v3 | 标准 cron 表达式 |
| WebSocket | gorilla/websocket | Go 标准 WebSocket 库 |
| HTTP 框架 | 待定（Go 原生 or Gin） | |
| 日志 | 待定（zap or slog） | |
| 数据库 ORM | 待定（GORM or sqlx） | |

> **注**：adshao/go-binance 已完整支持 Binance Futures REST + WebSocket，
> 可直接使用或参考其 HMAC 签名实现，降低手写封装工作量。

## DeepSeek R1 集成

- **接口**：OpenAI 兼容，`base_url = "https://api.deepseek.com"`
- **模型**：`deepseek-reasoner`（R1 推理）
- **定价**：约 $0.55~2.19 / M tokens
- **每次分析消耗**：Top20 标的数据卡片 + 新闻 ≈ 5000~8000 tokens

### LLM 在量化交易中的定位（行业共识）

- **适合**：基本面研究、定性分析增强、情绪信号提取、多维度信息综合
- **不适合**：替代数值化量化模型做最终决策
- **最佳实践**：作为信号确认层，而非独立信号源；要求 60%+ 置信度才执行

## 风控模块

### 硬规则（代码写死，不允许 R1 覆盖）

```
MAX_LEVERAGE      = 10      # 杠杆上限
MAX_SINGLE_POS    = 0.20    # 单仓不超过总资金 20%
MAX_TOTAL_RISK    = 0.50    # 总风险敞口不超过 50%
STOP_LOSS_PCT     = 0.05    # 强制止损 5%
MIN_LIQ_DISTANCE  = 0.15    # 距强平价至少 15%
```

### 补充风控机制（来自行业调研）

| 机制 | 说明 |
|------|------|
| **日亏损熔断** | 当日亏损超过总资金 X% 时自动停止交易，防止闪崩连锁亏损 |
| **资金费率侵蚀预警** | 持仓期间累计资金费率超阈值时提醒/强制平仓 |
| **相关性检测** | 多标的持仓时监控 BTC-ALT 相关性，高相关时自动缩减仓位 |
| **滑点预算** | 下单时预留 0.1~0.3% 滑点空间，避免实际成交价偏离 |
| **订单状态机** | Created→Submitted→Working→Filled/Cancelled 完整生命周期管理 |
| **逐仓模式** | 使用 Isolated Margin 而非 Cross Margin，防止单笔亏损波及全部资金 |

### 订单状态机

```
Created → Submitted → Accepted → Working
                                    │
                         ┌──────────┼──────────┐
                         ↓                     ↓
                  Partially Filled      Fully Filled（终态）
                         │
                         ↓
                    Cancelled（终态）
```

所有主流框架均实现此状态机，是合约交易系统的标配组件。
